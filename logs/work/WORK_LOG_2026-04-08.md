Known limitations:

- BM25 exact-match performance is currently validated on a small hand-built lexical slice, not a large real benchmark
- The implementation uses a simple identifier boost on top of BM25Okapi; this is pragmatic for exact-match precision but should be measured carefully on larger corpora
- Save/load currently stores documents and tokenized corpus, then rebuilds `BM25Okapi` on load
- Dense, KG, and Hybrid retrieval are still lightweight placeholders from the scaffold phase
- The smoke eval answer generator is still a simple evidence echo, not a final LLM-based answer generator

Concepts and topics used:

- BM25: a lexical ranking method that rewards term matches and rare terms
- BM25Okapi: the implementation from the `rank-bm25` package
- Tokenization: splitting text into searchable tokens
- Normalization: making tokens consistent, mainly by lowercasing and preserving useful identifier punctuation
- Exact-match identifiers: formal strings where punctuation and numbers matter, such as `164.512` or `torch.nn.CrossEntropyLoss`
- Near-match distractors: similar records used to test whether the correct exact match outranks close but wrong alternatives
- Top-k retrieval: returning the highest-scoring `k` results
- Save/load artifacts: persisting retriever state so the index can be reused
- Lexical benchmark slice: a small evaluation set focused on exact-match retrieval behavior
